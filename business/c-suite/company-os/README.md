# company-os

## 用途

設計並記錄公司的作業系統——包括使命、價值觀、原則、決策權限與文化規範，建立組織在規模擴展時的行為基礎與運作基準。

## 觸發方式

當使用者需要定義、記錄或重新設計公司的營運模式、治理體系或決策原則時觸發，例如：「幫我建立公司 OS」、「我們需要記錄決策方式」、「設計我們的營運原則」。

## 使用步驟

1. **釐清範圍** — 確認是建立完整的公司 OS（使命→價值觀→原則→決策權→儀式），還是針對特定層次。
2. **建立基礎** — 定義使命（為何存在）、願景（未來目標狀態）、價值觀（3–6 條行為承諾及定義）。
3. **制定營運原則** — 將價值觀轉化為具體的決策啟發，附上聲明、理由與實例，並標示原則衝突的解決方式。
4. **規劃決策權限** — 按策略／營運／戰術分類，套用 RACI 或 RAPID 框架，標示應下放或上報的決策。
5. **設計營運儀式** — 訂定節奏（每日、每週、每月、每季），規劃資訊流動與回饋循環。
6. **產出公司 OS 文件** — 使用結構化 Markdown 格式，每節包含目的、內容、負責人及審查週期。
7. **識別缺口與後續行動** — 列出缺失或不一致之處，並排序需要領導層對齊的決策清單。

## 規則

### 必須

- 每條原則必須對應公司實際面臨的取捨
- 為決策權限的每個類別提供具體範例
- 標示價值觀衝突並提供解決啟發
- 保持文件可快速掃讀——高層主管應能在 15 分鐘內讀完
- 為文件標記版本號（v1.0、日期）並指定負責人
- 文件包含定期審查與更新節奏

### 禁止

- 產出缺乏行為定義的空泛格言（如「我們重視誠信」）
- 在不了解當前人數與發展階段的情況下規定組織架構
- 將公司 OS 視為靜態文件——務必納入更新機制

## 範例

### 正確用法

使用者：「我們是一家 120 人的 SaaS 公司，牆上貼著價值觀但沒人使用。幫我建立真正的公司 OS。」

回應：先診斷「牆上價值觀」失效的原因，引導重新撰寫 4 條附有行為定義的價值觀，規劃 3 個層次的決策權限並附上 SaaS 運營的具體案例，產出完整的公司 OS 文件，包含每季審查儀式。

### 錯誤用法

使用者：「建立我們的公司 OS。」

回應：輸出一份通用的企業價值觀清單（創新、誠信、以客戶為中心），沒有行為定義、沒有決策權限、沒有與公司實際情境的連結。
