# context-engine

## 用途

建立結構化、可重複使用的策略背景文件，使 Claude 能在多個工作階段中持續擔任知識豐富的 C 級顧問角色。

## 觸發方式

當使用者希望將 Claude 設置為持續性策略顧問、在開始策略工作前提供公司背景，或建立可重複使用的背景文件時觸發，例如：「建立我的 context engine」、「幫我向 Claude 說明我的公司」、「建立一份公司背景文件」。

## 使用步驟

1. **訪談收集背景（若未預先提供）** — 針對 6 個領域提出結構化問題：商業模式與單位經濟、市場與競爭地位、組織設計與團隊、當前策略優先事項（前 3 項）、未來 90 天待做的關鍵決策，以及高層主管希望 Claude 的顧問風格與應對方式。
2. **建構背景文件** — 依照標準範本（見輸出格式）進行組織整理。
3. **驗證完整性** — 標記關鍵缺口，必要時提出釐清問題再定稿。
4. **設定顧問參數** — 偏好的溝通風格、希望 Claude 挑戰或支持的領域，以及顧問範圍外的主題。
5. **產出 Context Engine 文件** — 以 Markdown 格式呈現，設計為可貼入任何新工作階段開頭的格式。
6. **測試背景文件** — 透過一個示範策略問題驗證背景文件能否產出相關且個人化的回應。

## 規則

### 必須

- 捕捉策略優先事項背後的「為什麼」，而不只是「是什麼」
- 讓文件具備自足性——無需前置知識即可閱讀
- 包含版本日期，以便偵測文件是否過期
- 保持總字數在 600 字以內，確保快速載入背景

### 禁止

- 在設計為重複使用的文件中加入敏感財務細節（如精確 ARR、股權結構）
- 過度設計範本——必須快速填寫且快速閱讀
- 將背景文件視為永久性——應提示高層主管每季更新

## 範例

### 正確用法

使用者：「我是一家 50 人 B2B SaaS 公司的 CEO。我想定期使用 Claude 作為策略顧問。幫我設置 context engine。」

回應：逐一詢問 6 個結構化問題，將答案整合成完整的背景文件，標示 2 個缺口，產出一份已設定顧問參數的即用型背景文件。

### 錯誤用法

使用者：「建立我的 context engine。」

回應：要求使用者提供公司描述後，產出一段沒有結構、沒有顧問參數、沒有使用說明的 2 段落摘要。
