# decision-logger

## 用途

記錄策略性與營運性決策的完整背景、決策理由、考慮過的替代方案與審查觸發條件，使組織能從過去的選擇中學習，並對決策品質進行稽核。

## 觸發方式

當使用者希望記錄決策、回顧過往決策或建立決策記錄系統時觸發，例如：「記錄這個決策」、「幫我記錄我們選擇 X 的原因」、「建立決策登記冊」、「我需要記錄我們的策略轉向」。

## 使用步驟

1. **擷取決策聲明** — 一句清楚明確的句子：「我們決定於 [日期/條件] 採取 [行動]。」確認範圍：策略性、營運性或戰術性？
2. **記錄背景** — 是什麼問題或機會觸發了這個決策？當時的業務狀態？存在哪些限制條件？
3. **記錄考慮過的替代方案** — 列出 2–4 個曾認真評估的替代方案，每個附上描述、主要優點、主要缺點與棄用原因。
4. **陳述決策理由** — 選擇此路徑的主要原因、背後的關鍵假設，以及使用的資料或證據（並承認缺口）。
5. **識別利害關係人與決策過程** — 誰被諮詢（輸入）？決策後誰被告知？誰負責執行問責？
6. **定義後果與承諾** — 承諾或封鎖了哪些資源、關係或選項？對其他團隊或決策創造的相依性？可逆性評估。
7. **設定審查觸發條件** — 時間性：何時審查此決策？條件性：哪些訊號會觸發重新考量？
8. **指定決策 ID** — 格式：`DEC-YYYY-MM-NNN`，並標記類別。

## 規則

### 必須

- 明確區分事實與假設
- 在決策當下記錄，而非事後追記
- 記錄可逆性——單向門需要更高的記錄嚴謹度
- 至少包含 2 個真實的替代方案，而非稻草人
- 指定唯一的決策 ID 以便參考

### 禁止

- 為了讓決策在事後看起來更好而美化記錄
- 遺漏當時已承認的異議或重大風險
- 建立過於複雜的記錄格式，導致高層主管停止使用
- 將決策日誌與專案計畫或行動追蹤器混淆

## 範例

### 正確用法

使用者：「我們剛決定關閉中小企業產品線，全力專注企業市場。幫我記錄這個決策。」

回應：產出完整的 DEC 記錄，包含策略背景、3 個替代方案（保留中小企業、分拆、精簡運作）、決策理由（單位經濟、客服成本）、關鍵假設（企業管道健康度）、承諾（中小企業客戶通知、2 名 PM 重新部署），以及與企業 ARR 里程碑掛鉤的 6 個月審查觸發條件。

### 錯誤用法

使用者：「記錄我們轉型企業市場的決策。」

回應：「決策：轉型企業市場。日期：今天。原因：更好的經濟效益。」——沒有替代方案、沒有假設、沒有審查觸發條件、沒有利害關係人。
