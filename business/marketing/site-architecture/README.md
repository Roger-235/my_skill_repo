# site-architecture

## 用途

設計或稽核網站資訊架構，涵蓋 URL 結構、內部連結策略與爬蟲層級，以最大化 SEO 權重分配效率、提升使用者導航體驗，並確保搜尋引擎能有效發現與收錄所有重要內容。

## 觸發方式

當使用者需要規劃新網站架構、重新組織現有導航、稽核內部連結、降低爬蟲深度，或規劃 URL 層級結構時觸發。

## 使用步驟

1. 清點所有內容類型，對應至使用者旅程階段（認知、考慮、決策）。
2. 定義頂層分類法：反映使用者心智模型與關鍵字群集的主要類別。
3. 設計 URL 層級：根層 → 類別 → 子類別 → 頁面，重要頁面盡量維持在距首頁 3 次點擊以內。
4. 規劃支柱-群集內容模型：識別支柱頁面（廣泛主題）與群集頁面（具體子主題），建立雙向連結。
5. 設計導航：主導航、次導航、頁尾導航及麵包屑結構。
6. 制定內部連結規則：從高權威頁面到目標頁面的情境連結，以及錨點文字指引。
7. 稽核爬蟲效率：識別孤立頁面、內部連結不足的頁面及爬蟲陷阱。
8. 定義重新導向策略，包含任何 URL 變更的完整 301 對應表。
9. 建立 XML Sitemap 結構與 priority/changefreq 建議。

## 規則

### 必須

- 重要頁面距首頁維持在 3 次點擊以內
- 確保每個頁面至少有 3 條相關內部連結指向它
- URL Slug 使用描述性、與關鍵字相關的格式（小寫、連字符、去除停用詞）
- 超過 2 層層級的網站需加入麵包屑導航

### 禁止

- 在無充分理由下建立超過 5 層深度的 URL 結構
- 以查詢字串參數作為可收錄頁面的主要 URL 結構
- 讓類別頁面缺少連結至其子頁面的內部連結
- 設計將高轉換率頁面深埋於導航中的架構

## 範例

### 正確用法

「支柱頁：/project-management/ 連結至群集頁 /project-management/templates/、/project-management/software/、/project-management/for-teams/——所有群集頁均回連支柱頁，並在相關時互相連結。」

### 錯誤用法

「所有部落格文章使用 /post?id=12345 格式，無任何類別結構，相關文章之間也無內部連結。」
