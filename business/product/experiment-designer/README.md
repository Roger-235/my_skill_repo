# experiment-designer

## 用途

設計嚴謹的產品實驗，包括 A/B 測試、功能旗標（Feature Flag）計畫與假設驅動的驗證方案，提供明確的成功指標與統計方法指引。

## 觸發方式

當使用者需要以下協助時觸發：
- 設計 A/B 測試或分組實驗
- 驗證產品假設
- 定義實驗的成功指標
- 設定功能旗標實驗計畫

## 使用步驟

1. 將假設重新框架為：「我們相信 [改變] 將對 [受眾] 產生 [結果]，因為 [理由]」
2. 定義主要指標（決定成敗的單一可測量結果）
3. 定義護欄指標（不得惡化的次要指標）
4. 說明實驗變體（控制組 A 與處理組 B、C...）
5. 計算所需樣本量（MDE、基準轉換率、α=0.05、power=0.80）
6. 確定實驗執行時長（涵蓋完整業務週期）
7. 指定分組方式（隨機化單位：使用者、Session、裝置）
8. 制定分析計畫（統計檢驗方法、單尾/雙尾、p 值閾值）
9. 記錄風險（新奇效應、網路效應、季節性偏差）
10. 定義上線/不上線決策標準（實驗前預先登記）

## 規則

### 必須
- 在實驗開始前定義上線/不上線標準（預先登記）
- 至少包含一個護欄指標
- 明確陳述最小可偵測效果（MDE）
- 使用與假設匹配的隨機化單位

### 禁止
- 在達到所需樣本量之前建議停止實驗
- 在單一變體中測試多個不相關的改變
- 以頁面瀏覽量作為使用者行為假設的隨機化單位
- 在看到結果後才定義成功指標（p-hacking）

## 範例

### 正確用法
```
假設：我們相信在定價頁顯示社交認同（評論數量）將提高新訪客的試用註冊率，因為不確定性是主要障礙。

主要指標：試用註冊率（基準 4.2%）
護欄指標：付費轉換率不得下降超過 5%（相對值）
MDE：0.5pp 絕對提升（12% 相對提升）
所需樣本：每個變體約 8,400 人 | 執行時長：14 天
決策：若 p<0.05 且護欄指標維持則上線；否則繼續迭代。
```

### 錯誤用法
```
來測試新的按鈕顏色吧。跑幾天看看效果如何。如果點擊率上升就上線。
```
