# product-discovery

## 用途

引導產品團隊進行結構化的產品探索，整合使用者研究、定義機會空間、框架問題，並運用機會解決方案樹（OST）與 Jobs-to-be-Done 框架生成經過驗證的解決方案假設。

## 觸發方式

當使用者需要以下協助時觸發：
- 為新產品或功能啟動探索流程
- 整合研究發現
- 框架問題空間
- 建立機會解決方案樹（OST）

## 使用步驟

1. 錨定目標成果（探索所對應的產品或業務目標）
2. 繪製使用者旅程（JTBD 主任務）
3. 識別機會空間（從研究訊號中提取未滿足需求、痛點與渴望，歸類為主題）
4. 以需求陳述框架機會：「[使用者族群] 需要一種方式 [任務]，但 [當前障礙]」
5. 建立機會解決方案樹（OST）：成果 → 機會 → 解決方案 → 實驗
6. 依據頻率、嚴重性、當前滿意度與策略契合度排列機會優先順序
7. 為前 2–3 個機會各產生 3 個以上解決方案方向
8. 繪製假設風險地圖（每個解決方案的最高風險假設）
9. 設計探索實驗（訪談、原型、禮賓式測試、技術驗證）
10. 撰寫探索簡報（已選機會、理由、假設、下一步實驗）

## 規則

### 必須
- 在生成解決方案之前，先完成機會框架
- 以研究證據支撐每個機會，並標記來源
- 每個解決方案至少包含一個「可欲性」假設和一個「可行性」假設
- 在選擇解決方案前，先依機會吸引力排序

### 禁止
- 在繪製機會空間之前就跳到解決方案
- 將機會寫成解決方案（「使用者需要儀表板」是解決方案，不是機會）
- 跳過假設風險地圖——這能避免浪費開發週期
- 將使用者期望成果與業務成果混淆

## 範例

### 正確用法
```
機會：中型市場財務經理需要一種方式整合來自多個來源的費用資料，但目前每週花費 4 小時以上手動匯出和對帳試算表。

解決方案假設：
1. 與前 5 大費用工具的自動同步整合
2. 帶有欄位對應 UI 的 CSV 匯入精靈
3. 非結構化費用郵件的 AI 自動分類
```

### 錯誤用法
```
探索發現：使用者想要更好的儀表板。解決方案：建立有更多篩選器的新儀表板。
```
