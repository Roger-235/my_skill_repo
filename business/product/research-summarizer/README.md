# research-summarizer

## 用途

將使用者研究、訪談筆記、問卷資料和質性發現整合為結構化洞察、主題與可行的產品建議，讓團隊能立即採取行動。

## 觸發方式

當使用者需要以下協助時觸發：
- 整合訪談筆記或問卷結果
- 從可用性測試發現中提煉洞察
- 進行研究主題親和圖（Affinity Mapping）
- 從研究資料中提取可行的產品建議

## 使用步驟

1. 確認研究問題（研究設計要解答什麼）
2. 從輸入資料中提取原始觀察與引述
3. 進行親和群集（將觀察由下而上歸納為主題）
4. 命名主題（以需求或行為為基礎，而非功能名稱）
5. 評估主題強度（出現頻率與情感強度）
6. 整合洞察：「我們發現 [使用者族群] [行為/信念]，因為 [根本原因]，這意味著 [啟示]」
7. 識別 Jobs-to-be-Done（功能性、情感性、社會性任務）
8. 標記令人驚訝的發現（挑戰先前假設的洞察）
9. 產生建議（每個主要洞察對應一個產品行動與優先順序）
10. 輸出研究摘要報告

## 規則

### 必須
- 每個洞察至少附有一句直接引述作為佐證
- 報告頻率（多少位參與者，不只是「一些使用者」）
- 區分觀察（說了/做了什麼）與詮釋（代表什麼意義）
- 明確標記令人驚訝的發現——它們是最有價值的發現

### 禁止
- 用功能需求清單取代洞察
- 在沒有頻率或代表性背景的情況下報告發現
- 捏造引述或將改寫引述呈現為直接引述
- 在洞察之前先列出解決方案

## 範例

### 正確用法
```
主題：協作過程中對資料遺失的恐懼（n=7/10）
引述：「我在分享前總是複製一份檔案——我之前被坑過。」— P4
洞察：財務使用者將複製檔案作為安全行為，因為他們不信任即時協作工具，這意味著我們的衝突解決 UX 必須讓使用者清楚看見，而不只是讓它可靠。
建議：在共享檔案上加入可見的「版本歷史」指示器。優先順序：高。
```

### 錯誤用法
```
幾位使用者提到他們想要更好的協作功能。我們應該建立一個協作功能。使用者也說介面有點讓人困惑。
```
