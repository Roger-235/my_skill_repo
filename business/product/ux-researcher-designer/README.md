# ux-researcher-designer

## 用途

規劃並執行以使用者為中心的研究與設計活動，包含可用性測試設計、資訊架構、使用者流程圖與互動設計規格，降低可用性風險並設計符合真實使用者心智模型的介面。

## 觸發方式

當使用者需要以下協助時觸發：
- 規劃可用性研究或使用者測試
- 設計使用者流程圖
- 建立資訊架構（IA）
- 規格化互動模式
- 進行現有產品的 UX 審計

## 使用步驟

1. 定義 UX 問題：「我們需要了解/改善 [使用者體驗]，針對 [使用者族群]，以便 [成果]」
2. 選擇研究方法（發現：訪談、日記研究；驗證：可用性測試、A/B；行為：分析、Session 錄影）
3. 撰寫研究計畫（目標、方法、參與者條件、篩選問題、時程、主持人指引大綱）
4. 設計主持人指引（暖場、任務情境（放聲思考法）、追問問題、結束語）
5. 繪製使用者流程圖（每個關鍵任務：入口 → 步驟 → 決策點 → 出口）
6. 建立資訊架構（卡片分類方法論、網站地圖、導航層次、標籤規範）
7. 規格化互動模式（每個關鍵互動：觸發 → 系統回應 → 反饋 → 下一狀態）
8. 進行啟發式評估（如審計現有介面）——依據 Nielsen 十大啟發式原則評分，嚴重性評分 0–4
9. 整合發現（可用性問題按頻率 × 嚴重性排序；快速改善 vs. 需要重新設計）
10. 撰寫設計建議報告（每個問題：現狀 → 問題 → 建議 → 範例）

## 規則

### 必須
- 可用性測試使用基於任務的情境，而非引導性問題
- 以嚴重性評分回報可用性問題以利排序
- 在最終確定導航前透過實際使用者進行卡片分類來驗證 IA
- 使用者流程圖中包含錯誤狀態和邊緣情境

### 禁止
- 問使用者「你喜歡這個嗎？」——觀察行為，探索意圖
- 在沒有頻率的情況下回報發現（有多少位參與者遇到此問題）
- 僅根據內部利害關係人偏好設計資訊架構
- 在使用者流程圖中省略邊緣情境（空狀態、錯誤狀態、載入狀態）

## 範例

### 正確用法
```
任務情境：「您剛收到一封電子郵件，說您團隊的一份費用報告需要修改。請向我展示您接下來會怎麼做。」
（避免：「請前往費用報告區塊並點擊編輯。」）

發現：8 位參與者中有 6 位找不到「需要修改」篩選器
嚴重性：3（重大可用性問題）
建議：將篩選器移至清單視圖頂部；在導航項目上加入徽章計數。
```

### 錯誤用法
```
任務：「找到費用報告功能並告訴我們您是否喜歡它。」
發現：使用者覺得 UI 有點令人困惑。建議：讓它更清晰。
```
