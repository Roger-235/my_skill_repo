# atlassian-admin

## 用途

指導 Atlassian 管理員配置、治理並優化 Jira 和 Confluence 實例，涵蓋使用者管理、權限方案、空間架構與組織治理政策，支援可擴展且安全的團隊使用。

## 觸發方式

當使用者需要以下協助時觸發：
- 設定 Jira 或 Confluence 專案/空間
- 管理使用者與群組
- 配置權限方案
- 建立 Atlassian 治理政策
- 規劃 Atlassian 組織架構

## 使用步驟

1. 評估現有設定（專案結構、使用者數量、痛點）
2. 設計使用者與群組結構（依角色對齊的群組）
3. 配置權限方案（最小權限原則）
4. 設計通知方案（依角色與操作對齊；減少通知雜訊）
5. 配置問題類型方案與工作流程（跨相似專案類型標準化）
6. Confluence 空間治理（空間類型、權限模式、封存政策）
7. 建立命名規範（專案金鑰、空間金鑰、頁面標題、標籤分類）
8. 定義管理治理政策（建立專案/空間的權限、變更管理、升級路徑）
9. 規劃可擴展性（Atlassian Access：SSO/SCIM、審計日誌、資料駐留）
10. 記錄配置（生成治理文件與管理員操作手冊）

## 規則

### 必須
- 應用最小權限原則——預設為最低所需權限
- 使用專案角色而非個別使用者權限以利維護
- 跨專案類型標準化方案——避免每個專案自訂方案的蔓延
- 記錄每個自訂配置的業務理由

### 禁止
- 為只需要專案層級控制的使用者授予全域管理員權限
- 為每個專案建立新的權限方案——重用並擴展現有方案
- 在 Confluence 空間中為內部內容保留預設的「任何人」訪問權限
- 在未記錄轉換條件和驗證器的情況下配置工作流程

## 範例

### 正確用法
```
群組：「dev-team-leads」
權限：建立問題、編輯問題、管理衝刺、查看工程類別中的所有專案
理由：Team Lead 需要衝刺控制權但不需全域管理員；限定於工程專案範圍。

命名規範：專案金鑰 = [部門]-[團隊] 例如：ENG-PLATFORM、MKT-CONTENT
Confluence 空間金鑰：[團隊] 例如：PLATFORM、CONTENT
```

### 錯誤用法
```
給每個人管理員權限，這樣他們就能做任何需要做的事。為每個團隊建立新的權限方案，這樣更容易管理。
```
