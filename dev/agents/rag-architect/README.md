# rag-architect

## 用途

設計完整的檢索增強生成（RAG）系統，涵蓋文件擷取、切塊策略、嵌入模型選擇、向量資料庫、檢索管道、重排序與答案生成，並輸出可直接實作的架構文件。

## 觸發方式

當使用者要求：
- 設計或建構 RAG 系統或文件問答系統
- 選擇嵌入模型、向量資料庫或切塊策略
- 改善知識庫或語意搜尋的檢索品質
- 觸發關鍵詞：「design RAG」、「build RAG」、「retrieval-augmented generation」、「RAG architecture」、「語意搜尋管道」、「文件 Q&A」

## 使用步驟

1. 分析語料庫（文件類型、數量、更新頻率、語言）
2. 設計文件擷取與清理管道
3. 選擇切塊策略（固定大小、遞迴字元、語意、文件結構）
4. 選擇嵌入模型並說明理由
5. 選擇向量資料庫並說明理由
6. 設計檢索管道（查詢轉換、相似度指標、Top-K、元資料過濾）
7. 設計重排序策略
8. 設計答案生成（提示模板、上下文組裝、引用策略）
9. 規劃評估計畫（指標、工具、目標值）
10. 輸出完整 RAG 架構文件

## 規則

### 必須
- 必須指定切塊重疊大小，防止邊界上下文丟失
- 必須說明嵌入模型與向量資料庫的選擇理由
- 超過一萬份文件的語料庫必須納入重排序步驟
- 必須定義評估計畫
- 必須說明無相關上下文時的處理方式

### 禁止
- 禁止在不考慮文件結構的情況下推薦固定切塊大小
- 禁止省略元資料擷取
- 禁止僅因流行度選擇向量資料庫
- 禁止複雜問題不使用查詢轉換
- 禁止省略幻覺（Hallucination）緩解策略

## 範例

### 正確用法

```
使用者：為包含 50,000 份 PDF 合約的法律文件問答聊天機器人設計 RAG 系統。
→ 輸出完整架構文件，含語料庫分析、切塊策略、嵌入模型、向量資料庫、檢索管道、重排序、答案生成與評估計畫。
```

### 錯誤用法

```
使用者：為法律文件設計 RAG 系統。
回應：使用 OpenAI embeddings 嵌入所有文件，存到 Pinecone，然後擷取 top-5 最相似的片段。
→ 錯誤原因：無切塊策略、無語料庫分析、無選擇理由、無重排序、無評估計畫、無幻覺緩解策略。
```
