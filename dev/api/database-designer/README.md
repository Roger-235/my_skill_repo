# database-designer

## 用途

設計完整的資料庫架構，涵蓋資料庫類型選型、資料模型、正規化等級、索引策略、分割、複寫拓撲與備份恢復策略，產出可直接實作的設計文件。

## 觸發方式

當使用者要求：
- 為新應用程式或功能設計或選擇資料庫
- 在關聯式、NoSQL、時序或其他資料庫類型之間做選擇
- 設計系統的資料模型、Schema 或儲存層
- 觸發關鍵詞：「design database」、「database architecture」、「choose database」、「design data model」、「relational vs NoSQL」、「design storage layer」

## 使用步驟

1. 分析存取模式（主要與次要讀寫模式、查詢複雜度）
2. 分類資料特性（結構化/半結構化、時序、文件、圖形、鍵值）
3. 選擇資料庫類型並說明理由
4. 設計資料模型（實體、關係、基數、屬性類型）
5. 應用正規化（確定等級；說明有意為之的反正規化）
6. 設計索引策略（主鍵、次級索引、複合索引、覆蓋索引）
7. 設計分割/分片策略
8. 設計複寫與可用性拓撲
9. 設計備份與恢復（頻率、保留、PITR、RTO/RPO 目標）
10. 輸出完整資料庫設計文件

## 規則

### 必須
- 必須針對具體存取模式說明資料庫類型選擇理由
- 必須說明主鍵策略（UUID / 自動遞增 / 複合鍵）的選擇理由
- 每個高頻存取模式必須有對應索引設計
- 必須指定 RTO 和 RPO 目標
- 必須明確標示有意為之的反正規化

### 禁止
- 禁止在未分析存取模式的情況下推薦資料庫
- 禁止使用 FLOAT/DOUBLE 儲存金額，必須使用整數（分）或 DECIMAL
- 禁止未說明語意即使用可為 NULL 的外鍵
- 禁止新增未說明用途的索引
- 禁止省略備份與恢復策略

## 範例

### 正確用法

```
使用者：為多租戶 SaaS 計費系統設計資料庫。約 50 萬租戶，每月生成發票。
→ 輸出完整設計：存取模式分析、選擇 PostgreSQL（含 Row-Level Security 多租戶方案）、完整 Schema、複合索引設計、金額用整數分儲存、月份範圍分割、讀取副本、PITR 備份策略。
```

### 錯誤用法

```
使用者：為計費系統設計資料庫。
回應：使用 MySQL，建立 tenants 和 invoices 資料表並設定外鍵，加索引提升效能。
→ 錯誤原因：無存取模式分析、無類型選擇理由、無 Schema 定義、無索引策略、無分割、無複寫方案、無備份策略。
```
