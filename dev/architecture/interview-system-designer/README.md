# interview-system-designer

## 用途

以專業方法論引導系統設計面試題的完整作答：需求釐清、容量估算、高層次設計、元件深入探討，以及取捨討論，產出面試即用的結構化回答。

## 觸發方式

當使用者要求：
- 作答或練習系統設計面試題
- 設計知名系統（Twitter、Uber、URL 短網址、Netflix 等）
- 逐步拆解系統設計問題
- 觸發關鍵詞：「system design interview」、「design a system」、「design Twitter」、「design Uber」、「design URL shortener」、「system design question」

## 使用步驟

1. 釐清功能與非功能需求（明確範圍內與範圍外）
2. 估算規模（DAU、讀寫 QPS、儲存、頻寬）
3. 定義 API（3–5 個核心端點）
4. 繪製高層次架構圖（Mermaid）
5. 設計資料模型（實體、Schema、儲存選擇理由）
6. 深入探討 2–3 個關鍵元件
7. 討論主要決策的取捨
8. 識別 10x 規模下的瓶頸並提出解法
9. 3–5 點摘要

## 規則

### 必須
- 必須先釐清需求，再進行任何設計
- 必須包含 QPS、儲存、頻寬的概算
- 必須在繪製元件前先定義 API
- 必須進行至少 2 個元件的深入探討
- 必須為每個主要決策說明取捨
- 必須討論 10x 規模擴展情境

### 禁止
- 禁止跳過需求釐清直接給設計
- 禁止省略容量估算
- 禁止提出沒有取捨說明的設計決策
- 禁止在最關鍵路徑上設計無故障處理的系統
- 禁止使用技術而不說明為何適合此規模與存取模式

## 範例

### 正確用法

```
使用者：設計一個 URL 短網址服務。
→ 輸出完整回答：需求（縮短、重導向、自訂別名、分析）、容量估算（QPS、儲存）、API 定義、架構圖、資料模型、快取策略深入探討、取捨分析，以及 10x 規模下的瓶頸。
```

### 錯誤用法

```
使用者：設計一個 URL 短網址服務。
回應：把 URL 存在資料庫裡，有人點短網址時查出來重導向，用雜湊生成短代碼。
→ 錯誤原因：無需求、無容量估算、無 API 定義、無元件圖、無資料模型、無深入探討、無取捨、無擴展考量。
```
