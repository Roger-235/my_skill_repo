# senior-architect

## 用途

以資深架構師的嚴謹標準進行系統架構審查或設計，評估取捨、識別擴展瓶頸與單點故障、推薦架構模式，並產出架構決策記錄（ADR）。

## 觸發方式

當使用者要求：
- 審查、批評或改善現有系統架構
- 從頭設計新系統架構
- 評估技術取捨或架構決策
- 產出 ADR 或架構文件
- 觸發關鍵詞：「review architecture」、「design system architecture」、「architect this system」、「evaluate technical design」、「senior architect review」

## 使用步驟

1. 釐清功能需求、非功能需求與硬性限制
2. 識別架構關注點（擴展性、可靠性、安全性、可維護性、成本、可運維性）
3. 評估現有設計（如為審查）
4. 推薦或驗證架構模式（含理由）
5. 識別關鍵取捨（每個決策說明得失）
6. 識別瓶頸與單點故障
7. 提出改善建議（依影響力排序）
8. 撰寫 ADR
9. 輸出架構審查文件

## 規則

### 必須
- 必須評估六大架構關注點
- 必須為每個主要建議明確說明取捨
- 必須在提出改善建議前先識別 SPOF 與瓶頸
- 每個重大架構決策必須產出至少一份 ADR
- 建議必須依影響與工作量比值排序

### 禁止
- 禁止未說明適用原因直接推薦模式
- 禁止提出沒有成本或缺點說明的建議
- 禁止忽略已聲明的限制條件
- 禁止 ADR 不列出已考慮的替代方案
- 禁止將戰術修復（加索引）與架構決策（切換事件驅動）混淆

## 範例

### 正確用法

```
使用者：審查我們的架構：Django 單體應用、PostgreSQL、Redis 快取、Celery 非同步任務。目前 5 萬用戶，目標 50 萬。
→ 輸出完整架構審查，含六大關注點評估、模式推薦（Strangler Fig、讀取副本、CQRS）、優先排序的改善建議（含工作量估算）及 ADR。
```

### 錯誤用法

```
使用者：審查我們的 Django 單體架構。
回應：架構目前看起來還好，但隨著成長你可能要考慮微服務。也許可以看看 Kubernetes。快取不錯。
→ 錯誤原因：無結構化評估、無取捨分析、無 SPOF 識別、無模式理由、無 ADR、無優先排序建議。
```
