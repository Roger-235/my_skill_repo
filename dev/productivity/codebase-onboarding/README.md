# codebase-onboarding

## 用途

為加入陌生程式碼庫的工程師產生結構化的入職指南，涵蓋架構概覽、關鍵入口點、資料流、開發慣例與第一次貢獻建議。

## 觸發方式

當使用者請求協助了解新的程式碼庫、需要儲存庫導覽、需要為新團隊成員準備入職材料，或詢問在陌生專案中從哪裡開始閱讀時觸發。

關鍵字：`onboarding`、`new codebase`、`understand codebase`、`codebase tour`、`first day`、`explore repo`、`codebase orientation`

## 使用步驟

1. **閱讀入口點檔案** — `README.md`、`CONTRIBUTING.md`、`docs/` 資料夾、頂層設定檔。
2. **對應技術堆疊** — 從套件清單識別語言、框架、建置工具、測試執行器與關鍵相依套件。
3. **追蹤主要入口點** — 找到 `main()`、`index.ts`、`app.py` 或 `cmd/` 等等效檔案，並追蹤啟動序列。
4. **識別架構層次** — 描述高層次結構（例如 API 層 → 服務層 → 儲存庫層 → 資料庫）。
5. **找出關鍵領域概念** — 識別 3–5 個核心領域模型/實體及定義它們的檔案。
6. **對應資料流** — 追蹤至少一個從入口到儲存的端對端請求或工作。
7. **強調慣例** — 命名慣例、目錄結構規則、錯誤處理模式、日誌方式與程式碼風格工具。
8. **列出本機開發指令** — 如何安裝相依套件、本機執行應用程式、執行測試與 lint。
9. **建議第一次貢獻** — 指向 `good-first-issue` 標籤、測試檔案或文件改進作為安全的起始點。

## 規則

### 必須

- 所有描述必須來自儲存庫中的實際檔案，不得憑空創造結構。
- 包含從實際 README 或 Makefile 驗證過的可執行本機開發指令。
- 架構描述保持在概念層次，避免逐行解釋程式碼。
- 標注文件不足或已知複雜度較高的區域。

### 禁止

- 禁止創造儲存庫中不存在的檔案路徑或模組名稱。
- 禁止在指南中包含設定檔中發現的憑證、API 金鑰或密鑰。
- 禁止略過本機開發快速入門——這是新進人員最需要的部分。

## 範例

### 正確用法

```markdown
## 技術堆疊
| 元件 | 技術 | 備注 |
|------|------|------|
| API  | Go 1.22 + Gin | REST，版本化於 /api/v1 |
| 資料庫 | PostgreSQL 15 | 使用 golang-migrate 進行遷移 |

## 本機開發快速入門
```bash
git clone https://github.com/org/repo && cd repo
cp .env.example .env
docker compose up -d db redis
make run   # 在 :8080 啟動
make test  # 執行單元與整合測試
```
```

### 錯誤用法

```markdown
## 架構
程式碼庫使用典型的 MVC 模式。
（沒有具體檔案路徑，沒有儲存庫中的實際依據——過於模糊）

## 技術堆疊
前端可能使用 React。
（猜測，未從儲存庫驗證）
```
