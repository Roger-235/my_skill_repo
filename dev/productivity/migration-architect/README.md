# migration-architect

## 用途

設計並執行技術遷移——包含資料庫結構遷移、框架升級、語言版本升版、基礎設施平台遷移與資料遷移——並附帶回滾計畫與零停機策略。

## 觸發方式

當使用者需要遷移資料庫結構、升級框架或語言版本、將基礎設施移至新平台，或重組資料儲存庫時觸發——特別是在需要零停機或向後相容性的情況下。

關鍵字：`migration`、`migrate`、`upgrade`、`database migration`、`schema change`、`platform migration`、`zero downtime`、`expand-contract`

## 使用步驟

1. **定義範圍與風險** — 列舉變更內容、受影響的服務，以及風險等級（資料遺失可能性、停機時間、消費者數量）。
2. **選擇遷移策略**：
   - **展開-收縮（Expand-Contract）**（零停機結構變更的首選）：新增欄位/資料表 → 遷移資料 → 更新程式碼 → 移除舊欄位/資料表。
   - **藍綠部署**：啟動含遷移後狀態的新環境，切換流量，保留舊環境作為回滾。
   - **維護視窗內就地升級**：適用於非關鍵系統或小型資料集。
   - **絞殺者無花果（Strangler Fig）**：用於平台遷移，逐步替換舊系統元件。
3. **撰寫遷移計畫** — 含操作、指令/腳本、驗證檢查，以及每個步驟的回滾步驟。
4. **處理向後相容性** — 確保程式碼在轉換期間同時支援新舊狀態（雙寫、功能旗標、API 版本控制）。
5. **測試遷移** — 對接近生產環境的資料副本執行；測量持續時間，驗證資料完整性。
6. **定義回滾觸發條件** — 指定觸發或建議回滾的可觀察條件（錯誤率飆升、資料不一致計數）。
7. **分階段執行** — 大型資料遷移分批處理，限制速率以避免鎖定或壓垮資料庫。
8. **驗證並清理** — 遷移後執行資料完整性檢查；移除相容性墊片和已棄用的程式碼。

## 規則

### 必須

- 任何破壞性遷移步驟前，一律備份（並驗證備份）。
- 對活躍資料庫的零停機結構變更使用展開-收縮模式。
- 測試回滾程序，而不只是正向遷移。
- 大型資料遷移分批處理（每批 1,000–10,000 筆），並設定可配置的批次大小。
- 在遷移計畫中包含資料完整性驗證查詢。

### 禁止

- 禁止在同一次部署中同時刪除欄位/資料表和移除使用它們的應用程式程式碼。
- 禁止在生產環境中未備份就執行結構遷移。
- 禁止在尖峰流量期間對大型資料表使用 `ALTER TABLE ... LOCK=EXCLUSIVE`。
- 禁止在原始資料來源仍在接收寫入的情況下，未使用 CDC 或雙寫就執行平台遷移複製資料。

## 範例

### 正確用法

```sql
-- 展開-收縮：第 1 階段——新增欄位（向後相容）
ALTER TABLE users ADD COLUMN display_name VARCHAR(255);

-- 第 2 階段——分批填充
UPDATE users SET display_name = first_name || ' ' || last_name
WHERE display_name IS NULL AND id BETWEEN :start AND :end;

-- 第 4 階段——舊欄位不再被程式碼參照後才刪除
ALTER TABLE users DROP COLUMN first_name, DROP COLUMN last_name;
```

### 錯誤用法

```sql
-- 同一次部署中同時刪除程式碼參照和欄位——造成停機
-- 對 1000 萬列資料表進行無分批更新——鎖定整個資料表，造成逾時
UPDATE users SET display_name = first_name || ' ' || last_name;

-- 破壞性操作前未備份
ALTER TABLE orders DROP COLUMN legacy_status;
```
