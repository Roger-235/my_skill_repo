# Build Fix — 建置錯誤修復

系統性診斷建置與編譯失敗的根本原因，並套用精準修復，而非遮蔽錯誤。

## 功能

- 解析建置錯誤輸出，提取檔案路徑、行號、錯誤代碼與錯誤訊息
- 分類錯誤類型：語法錯誤、型別錯誤、依賴錯誤、連結器錯誤、設定錯誤、環境問題
- 追溯根本原因，而非只處理錯誤訊息出現的那一行
- 套用最小範圍的修復，確認建置成功後才結束
- 多個錯誤時，從第一個開始處理（後續錯誤往往是連鎖反應）

## 觸發時機

**手動觸發**：
- 「build fails」、「compilation error」、「can't build」、「build broken」
- 「linker error」、「type error at build time」
- 「建置失敗」、「編譯錯誤」、「build 出錯」、「無法編譯」

**不觸發**：
- 執行時錯誤或測試失敗 → 改用 debug
- 不阻擋建置的 lint 警告

## 使用方法

提供建置指令與完整的錯誤輸出，skill 會按照以下步驟處理：

1. **解析錯誤** — 從輸出中提取關鍵資訊（檔案、行號、錯誤類型）
2. **分類錯誤** — 判斷是語法、型別、依賴、連結器、設定還是環境問題
3. **找出根本原因** — 追溯錯誤的真正來源，而非症狀所在的位置
4. **套用修復** — 針對根本原因做最小範圍的更動
5. **重新建置** — 執行相同的建置指令確認修復成功
6. **若仍失敗** — 以新的錯誤輸出重複上述流程

**輸出格式範例**：

```
## Build Fix: <建置指令>

**Error type:** <錯誤類型>
**Root cause:** <一行說明根本原因>
**Location:** <file:line>

### Error Output
<相關錯誤訊息>

### Fix Applied
<修改前後的 diff>

### Build Result
✓ Build succeeded  /  ✗ Still failing — see new error above
```

## 前置需求

- 失敗建置指令的完整錯誤輸出
- 所使用的建置指令（例如 `npm run build`、`go build ./...`、`cargo build`）

## 安全性

- 不使用 `@ts-ignore`、`#[allow(...)]`、`// nolint` 等指令來遮蔽錯誤
- 不刪除測試檔案來讓建置通過
- 不修改 `.gitignore` 或建置設定來隱藏問題
- 不在沒有修復根本原因的情況下新增 `any` 型別來消除 TypeScript 錯誤
