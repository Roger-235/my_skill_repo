# dependency-auditor

## 用途

審計專案依賴的安全漏洞、過時版本、授權風險與供應鏈問題，並產出優先排序的修復計畫。

## 觸發方式

當使用者要求審計依賴、檢查有漏洞或過時的套件、審查授權合規性，或評估專案供應鏈安全時觸發。

## 使用步驟

1. 解析依賴清單，區分生產與開發依賴
2. 安全漏洞檢查——對照 CVE 資料庫，標注 Critical/High 漏洞與安全版本
3. 版本時效性檢查——識別落後主要版本、廢棄套件
4. 授權審計——識別 SPDX 授權識別碼，標注授權衝突風險
5. 供應鏈風險評估——少量下載者、近期轉讓所有權、疑似仿冒名稱
6. 生成按嚴重性排序的修復計畫
7. 提供升級所有 Critical/High 發現的精確套件管理器指令

## 規則

### 必須
- 區分生產與開發/測試依賴的嚴重性——僅開發環境的漏洞優先級較低
- 每個安全發現包含特定的 CVE ID 與 CVSS 分數
- 為每個有漏洞的套件提供精確的安全版本
- 每個 Critical 與 High 發現包含升級指令
- 標注可能造成法律暴露的授權風險

### 禁止
- 建議移除依賴而不提供替代方案或說明理由
- 將僅開發環境的 CVE 作為生產部署背景下的 Critical 問題回報
- 忽略傳遞性依賴——許多實際漏洞存在於傳遞性依賴中
- 虛構 CVE ID——不確定時注明「請驗證 OSV/NVD」
- 建議降級到同樣有 CVE 的舊版本

## 範例

### 正確用法

審查 package.json，標注 lodash 原型污染漏洞（CVE-2021-23337，High）、axios SSRF 漏洞（CVE-2021-3749，High）、jsonwebtoken 關鍵認證繞過（CVE-2022-23529，Critical），提供各套件的 npm install 指令，並標注廢棄套件與供應鏈風險。

### 錯誤用法

回應「你應該執行 npm audit 來檢查漏洞。確保保持依賴更新！」——無實際審計、無具體發現、無 CVE 引用、無修復計畫。
