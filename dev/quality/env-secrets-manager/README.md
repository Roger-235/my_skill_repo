# env-secrets-manager

## 用途

為應用程式與 CI/CD 管道設計並實施安全、實用的環境變數與 Secret 管理策略，防止憑證洩露，同時保持良好的開發體驗。

## 觸發方式

當使用者要求管理 Secret 或環境變數、在管道中保護憑證、設置 Secret Vault、輪替憑證，或防止 Secret 洩露時觸發。

## 使用步驟

1. 清點現有 Secret（資料庫憑證、API 金鑰、服務帳號、TLS 憑證、簽名金鑰）
2. 評估當前暴露情況（代碼庫中的 Secret、CI 日誌、過於廣泛的憑證）
3. 依環境設計 Secret 管理策略（開發/.env、CI/平台 Secret、生產/Vault）
4. 設計存取模式——應用程式啟動時從 Secret Manager SDK 讀取，遵循最小權限
5. 設計 Secret 輪替策略（零停機輪替模式）
6. 生成實作產物（.env.example、.gitignore、應用程式啟動代碼、預提交鉤子配置）
7. 針對已偵測到的暴露採取補救措施

## 規則

### 必須
- 將任何已確認的 Secret 暴露視為 Critical 事件——立即輪替
- 提供帶佔位符值的 .env.example，絕不包含真實值
- 建議在預提交鉤子和 CI 管道中進行 Secret 掃描
- 設計零停機 Secret 輪替
- 應用最小權限存取——應用程式只應讀取自己擁有的 Secret

### 禁止
- 在任何建議或生成的檔案中提交含真實值的 .env 文件
- 建議在源代碼的環境變數預設值中存儲 Secret
- 建議將 Secret 嵌入 Docker 映像或建構產物
- 建議停用 Secret 掃描以「疏通」管道
- 建議多個應用程式共用單一管理員憑證

## 範例

### 正確用法

使用者：「我在 GitHub 倉庫中發現了資料庫密碼，該怎麼辦？」

立即輪替密碼（視為已洩露）、審計 DB 存取日誌、使用 git filter-repo 從歷史中移除 Secret、設置預提交鉤子、將 DB 憑證遷移至 AWS Secrets Manager——並提供代碼片段與 .gitignore 條目。

### 錯誤用法

回應「只需刪除檔案並推送新提交以從最新版本中移除它。你也可以使用環境變數。」——不輪替憑證、不處理 git 歷史、不防止再次發生。
