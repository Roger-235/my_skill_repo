# pr-review-expert

## 用途

執行全面、可行的拉取請求審查，涵蓋正確性、安全性、效能、可維護性與測試覆蓋率，並提供具優先順序的結構化回饋。

## 觸發方式

當使用者要求審查拉取請求、審查差異、提供代碼審查意見，或在合併前評估變更時觸發。

## 使用步驟

1. 理解 PR 目的——閱讀描述與連結 Issue
2. 審查正確性——邏輯錯誤、邊界情況、錯誤處理
3. 審查安全性——注入風險、認證/授權繞過、敏感資料暴露
4. 審查效能——N+1 查詢、缺少索引、熱路徑中的低效操作
5. 審查可維護性——代碼清晰度、測試覆蓋率、DRY 原則
6. 分類發現（Blocking / Recommended / Nit）
7. 撰寫結構化審查報告

## 規則

### 必須
- 每個發現標注優先等級（Blocking / Recommended / Nit）
- 每個發現包含具體的檔案與行號引用
- 每個 Blocking 問題提供建議修復方向
- 承認 PR 的優點——審查應具建設性
- 安全問題一律標注為 Blocking

### 禁止
- 將 Blocking 安全問題降級為 Recommended
- 在沒有既定風格指南的情況下對風格發表評論
- 要求修改 PR 範圍之外的問題
- 在有未解決 Blocking 問題的情況下批准 PR
- 提供「這裡可以更好」等含糊回饋

## 範例

### 正確用法

審查新增 API 端點的差異，識別缺少認證中介軟體（Blocking）、SQL 字串拼接注入風險（Blocking）、缺少錯誤路徑測試（Recommended），以及變數命名（Nit）——每項均附具體行號與修復建議。

### 錯誤用法

回應「整體看起來不錯！也許多加一些測試，確保代碼整潔。」——無具體發現、無行號引用、無優先排序。
