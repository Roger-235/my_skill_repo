# tech-debt-tracker

## 用途

系統性地識別、分類與優先排序程式碼庫中的技術債務，並產出與業務影響對齊的結構化修復路線圖。

## 觸發方式

當使用者要求評估技術債務、審計程式碼庫健康狀況、建立重構計畫、建構債務清單，或了解遺留代碼成本時觸發。

## 使用步驟

1. 定義債務類別（架構、代碼品質、測試、依賴、文件、運營）
2. 掃描債務信號（過長函式、TODO/FIXME 注釋、缺乏測試、過時依賴）
3. 為每個債務項目評分（影響 × 風險 / 工作量）
4. 建構優先排序的債務清單
5. 設計修復路線圖（現在 / 下季 / 待辦）
6. 估算總工程成本，以「債務利息」呈現持續影響

## 規則

### 必須
- 為每個債務項目評估影響、風險與工作量
- 區分具有主動風險的債務與純粹美觀性的債務
- 以業務術語（速度影響、事故風險）而非工程美學呈現債務
- 包含帶時間範圍的修復路線圖
- 識別「債務利息」——不償還每個項目的持續成本

### 禁止
- 將每個不完美之處都列為債務——專注於實質影響速度或可靠性的項目
- 在沒有分階段遷移計畫的情況下建議完全重寫
- 將外觀性債務（命名、格式）優先於結構性或測試債務
- 生成沒有可行修復指導的債務清單
- 建立路線圖時忽略團隊容量

## 範例

### 正確用法

使用者：「評估我們支付服務的技術債務。」

生成跨 4 個類別的 15 項清單，評分每個項目，識別高風險項目為「立即」優先，估算總工程天數，以「每 sprint 速度稅」呈現持續影響。

### 錯誤用法

列出每個不具描述性的變數名稱和每個可以更整潔的函式——無評分、無優先排序、無業務影響框架、無路線圖。
