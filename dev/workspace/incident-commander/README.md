# incident-commander

協調端對端技術事故應對：從偵測與分類到緩解、解決，以及無責備事後分析。

## 用途

擔任事故指揮官：分類嚴重度、協調溝通、驅動緩解行動、確認解決，並撰寫可執行的事後分析報告。

## 觸發方式

- 「事故處理」、「生產故障」、「緊急事件」、「事後分析」、「障礙報告」
- incident, outage, production down, P0, P1, Sev1, Sev2, on-call
- incident commander, incident response, service degradation, postmortem

**不觸發：** 開發環境除錯（用 `debug`）、CI 建置失敗（用 `build-fix`）、無用戶影響的非生產問題。

## 使用步驟

1. 宣告並分類（嚴重度 P0/P1/P2，建立事故頻道，分配角色）
2. 評估影響（受影響用戶、功能、是否有資料損失）
3. 5 分鐘內溝通初始狀態
4. 收集信號（錯誤率、延遲、近期部署、設定變更）
5. 識別近因（從時間線找相關變更，建立假設）
6. 執行緩解（回滾、功能標誌、擴容、流量轉移）
7. 確認緩解（驗證指標恢復基準線）
8. 調查根本原因（影響緩解後深入調查）
9. 解決並溝通（指標穩定 ≥ 15 分鐘後宣告解決）
10. 撰寫無責備事後分析（48 小時內，含行動項目）

## 規則

### 必須
- 確認用戶影響後 5 分鐘內宣告事故
- 5 分鐘內發出第一份狀態更新
- 緩解影響先於調查根本原因
- P1 及以上事故必須撰寫書面事後分析
- 行動項目必須有負責人和到期日

### 禁止
- 在事後分析中指責個人
- 指標穩定少於 15 分鐘就宣告解決
- 在客戶面向狀態更新中分享未經確認的資訊

## 範例

### 正確用法

```
事故: INC-2847 — Checkout 服務 40% 請求返回 503
嚴重度: P1 | 宣告時間: 14:32 UTC | 指揮官: Alice
影響: ~40% 結帳失敗 | 資料損失: 否
假設: checkout-service v2.4.1 記憶體洩漏（14:15 UTC 部署）
行動: 14:38 啟動回滾至 v2.4.0，預計 5 分鐘
下次更新: 14:45 UTC
```

### 錯誤用法

「結帳好像壞了，可能是資料庫或新部署的問題，我們還在查，有消息再更新。」

無嚴重度分類、無指揮官、無量化影響、無時間線、無行動計畫、無承諾更新時間。
