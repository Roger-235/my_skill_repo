# self-improving-agent

## 用途

透過分析近期互動、識別錯誤模式，以及提出技能或工作流程的具體改進措施，實現結構化自我反思與效能提升。

## 觸發方式

當使用者要求 Claude 反思自身效能、分析近期互動中的失誤、提議改善其行為或技能，或對工作階段或任務進行回顧時觸發。

## 使用步驟

1. 收集反思材料——次優的輸出或決策、使用者回饋、需要回溯或修正的情況
2. 分類失敗模式（觸發不匹配、前置條件缺口、步驟遺漏、規則違反、輸出格式偏差、範圍錯誤、情境丟失）
3. 識別根本原因——技能設計問題、觸發問題、判斷問題還是偶發事件
4. 提出具體改進——草擬 SKILL.md 的具體文字變更，或精煉觸發關鍵字
5. 評估影響——每個提議改進能防止什麼失敗，以及意外副作用的風險
6. 提呈改進計畫——按影響力與可信度排序
7. 提議實作——詢問使用者要應用哪些改進；使用 `skill-edit` 進行 SKILL.md 變更

## 規則

### 必須
- 每個提議改進都基於具體觀察到的失敗——不作推測性改進
- 區分技能設計問題與一次性判斷錯誤
- 評估每個提議變更的風險（改進可能引入新問題）
- 對不確定性保持誠實——若根本原因不清楚，明確說明
- 優先處理反覆出現的模式而非孤立事件

### 禁止
- 提議使技能過於限制性以避免罕見邊緣情況的變更
- 以效能最佳化為由建議移除安全限制（Never 規則）
- 將改進呈現為確定的修復——所有改進都是待驗證的假設
- 以將責任歸咎於使用者的方式反思失敗
- 在未獲使用者輸入或情境的情況下生成改進

## 範例

### 正確用法

使用者：「你一直在沒有詢問雲端提供商的情況下生成 Terraform 代碼。你能反思一下嗎？」

識別前置條件缺口——跳過了詢問雲端提供商的步驟，預設為 AWS。根本原因：前置條件說明要「知道目標雲端提供商」，但若未知時未要求詢問。提議：在步驟 1 加入「若未說明雲端提供商，在生成任何代碼前先詢問」，並在 Must 規則中加入「若未說明目標雲端提供商則必須詢問——絕不預設為 AWS」。評估：高可信度能防止再次發生；副作用風險低。

### 錯誤用法

使用者：「你之前犯了一些錯誤。」

回應「我下次會努力做得更好！我應該更加小心和全面。」——無具體失敗分析、無模式識別、無具體改進提議、未提出技能變更。
